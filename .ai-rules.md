0) Purpose

This file defines non-negotiable rules for code generated or modified with AI in this repo.
If any instruction conflicts with this file, this file wins.

1) Architecture rules (modular monolith)

Backend is a modular monolith. No microservices.

Module layout

All backend features must live under:
apps/api/src/modules/<module>/{api,service,domain,repo}

Layering rule

api/ = validate + call service (no business logic)

service/ = orchestration / use-cases

domain/ = pure domain rules (no db, no http)

repo/ = db access only

Dependency rules (strict)

domain/ must not import from api/ or repo/

service/ may import domain/ + repo/

api/ may import service/ only

2) International-ready by design
i18n

No hardcoded user-facing strings in app/web/mobile.

All UI text uses i18n keys.

Backend returns keys, not translated sentences.

Locale + timezone + currency

Store all timestamps in DB as UTC (timestamptz).

Each salon has timezone, locale, currency.

Prices stored as amount (int minor units) + currency (ISO-4217).

Phone numbers

Store customer phone as E.164 (e.g. +4512345678).

3) API contract rules
OpenAPI is source of truth

API changes must update: docs/openapi-v1.yaml

API versioning: all endpoints under /v1/...

Standard error format (always)

All errors must follow:

{
  "code": "string.key",
  "message": "string.key",
  "details": {},
  "traceId": "uuid"
}

Backward compatibility

Never break existing fields or behavior in v1.

Only add optional fields or new endpoints.

4) Data & database rules (Supabase/Postgres)
Migrations are source of truth

All schema changes via supabase/migrations/*

Never “manual edit” production schema.

Booking overlap is enforced by DB

The database must prevent overlapping bookings per staff member.

Use Postgres exclusion constraint (range) or equivalent.

App-level checks are not enough.

Audit log required for critical actions

Must write audit entries for:

login/session changes

role changes

booking create/update/cancel

payment status changes

data export/delete actions

Soft delete

Use deleted_at for user/customer data where appropriate.

5) Security & safety rules (non-negotiable)
Auth & authorization

Role-based access: owner/admin/staff.

Every request must pass authorization checks.

Never trust client-provided salon_id without verifying membership.

Secrets & logging

Never commit secrets, tokens, private keys.

Never log: access tokens, refresh tokens, card data, passwords.

Logs must be structured (JSON) and include a request traceId.

Webhooks

Verify webhook signature (Stripe, etc.)

Webhook handlers must be idempotent.

Input validation

Validate all inputs at API layer using schemas.

Reject unknown fields where possible (avoid “silent acceptance”).

Rate limiting

Rate limit auth endpoints and webhook endpoints.

6) Payments rules
Provider abstraction

Payments must go through a provider interface:

PaymentProvider.createCheckout(...)

PaymentProvider.handleWebhook(...)

Status model

Booking status and payment status are separate.
Do not mark booking confirmed until payment success is verified.

7) Notifications rules (email/SMS)
Outbox pattern

All outgoing notifications must be written to notification_outbox first.
Workers/processors send them asynchronously.

Templates

Templates must use i18n keys and variables (no hardcoded language).

8) Testing rules
Critical flows must have integration tests

Minimum required integration tests:

booking overlap prevention

booking → checkout → webhook confirms booking

webhook idempotency (same event twice)

No “tests later”

If a change touches booking or payments, tests must be updated in the same PR.

Integration tests live in: apps/api/test/integration

9) Code quality rules

TypeScript strict mode must stay enabled.

ESLint + Prettier must pass.

No any unless justified and isolated.

No dead code paths.

Keep functions small and testable.

10) What AI should output

When asked to implement something, AI must provide:

the files to create/modify

minimal code changes

migration if schema changes

OpenAPI update if endpoint changes

tests if behavior changes

AI must not:

invent new providers/features not requested

implement out-of-scope features (see Project Constitution)
